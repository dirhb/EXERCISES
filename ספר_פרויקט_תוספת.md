# פרק 10: בדיקות המערכת (Testing)

## 10.1 מבוא לבדיקות תוכנה

בדיקות תוכנה (Software Testing) הן חלק בלתי-נפרד ממחזור חיי הפיתוח. מטרת הבדיקות היא לאמת (Verify) שהמערכת פועלת בהתאם לדרישות שהוגדרו בשלב הניתוח, ולאמת (Validate) שהמערכת עונה על צרכי המשתמש האמיתיים. ביצוע בדיקות שיטתיות מוקדם ככל האפשר חוסך עלויות גבוהות של תיקון באגים בשלבים מאוחרים.

בפרויקט **JobFindUltimate** בוצעו שלושה סוגי בדיקות עיקריים: בדיקות יחידה (Unit Tests), בדיקות אינטגרציה (Integration Tests) ובדיקות מקצה לקצה (End-to-End Tests). הבדיקות בוצעו ידנית לאורך כל שלבי הפיתוח, ולא בעזרת כלי Automation.

# פרק 11: אבטחת מידע (Information Security)

## 11.1 עקרונות אבטחת מידע

אבטחת מידע (Information Security) בפיתוח תוכנה מתבססת על שלושה עקרונות מרכזיים, המכונים **CIA Triad**:

**Confidentiality (סודיות):** הבטחה שמידע רגיש נגיש רק לגורמים מורשים. בפרויקט זה: נתוני משתמש (סיסמה, דוא"ל, שכר) נשמרים ב-Session ולא נחשפים בכתובות URL.

**Integrity (שלמות):** הבטחה שהנתונים לא שוּנו בצורה בלתי-מורשית. בפרויקט זה: Parameterized Queries מונעות שינוי נתונים על ידי SQL Injection.

**Availability (זמינות):** הבטחה שהמערכת זמינה למשתמשים מורשים. בפרויקט זה: ניהול חיבורי DB עם `finally { CloseConnection() }` מונע נעילה ממושכת.

## 11.2 איומים שזוהו ואמצעי הגנה

### איום 1: SQL Injection

**רמת סיכון:** גבוהה מאוד

**תיאור:** תוקף מזין קוד SQL זדוני בשדות קלט כדי לשנות שאילתות במסד הנתונים.

**דוגמה לניצול:**
```sql
-- קלט זדוני בשדה Username:
' OR '1'='1' --

-- שאילתה מסוכנת ללא הגנה:
SELECT * FROM [User] WHERE UserName='' OR '1'='1' --'
-- תוצאה: מחזירה את כל המשתמשים!
```

**אמצעי הגנה שמומש:**
```csharp
// כל שאילתה משתמשת ב-OleDbParameter:
string sql = "SELECT * FROM [User] WHERE UserName=@UserName";
this.helperOleDb.AddParameters("@UserName", userName);
// הערך מועבר כ-Parameter, לא כחלק מ-SQL text
// Access.OleDb מבצע Escaping אוטומטי של תווים מיוחדים
```

**סטטוס:** ✅ מוגן לחלוטין

### איום 2: Cross-Site Scripting (XSS)

**רמת סיכון:** בינונית

**תיאור:** תוקף מזין קוד JavaScript זדוני שמתבצע בדפדפן המשתמש.

**דוגמה לניצול:**
```html
<!-- אם תיאור משרה מוצג ללא Escaping: -->
JobDescription = "<script>document.cookie='hacked=true';</script>"
```

**אמצעי הגנה שמומש:**
- Razor Views ב-ASP.NET Core מבצעות **HTML Encoding אוטומטי** של כל ערך שמוצג דרך `@Model.Field` או `@ViewBag.Field`.
- התוצאה: `<script>` הופך ל-`&lt;script&gt;` – טקסט גלוי, לא קוד מבוצע.

**סטטוס:** ✅ מוגן על ידי מנגנון Razor המובנה

### איום 3: Broken Access Control (שבירת בקרת גישה)

**רמת סיכון:** גבוהה

**תיאור:** משתמש ממחלקה אחת ניגש לפונקציות של מחלקה אחרת (למשל: Employee מגיע ל-Employer/PostJob).

**אמצעי הגנה שמומש:**
```csharp
// בכל Action Method:
private bool IsAuthorized() => SessionHelper.IsEmployee(HttpContext.Session);

public async Task<IActionResult> Resume()
{
    if (!IsAuthorized()) return RedirectToAction("Home", "Guest");
    // ...
}
// SessionHelper.IsEmployee בודק: GetUserTypeID(session) == 2
// UserTypeID מאוחסן ב-Server-Side Session, לא ב-Cookie שהלקוח יכול לזייף
```

**מה שנשאר לשפר:** בקרי ה-API עצמם (JobWebService) אינם מבצעים אימות – כל מי שיודע את כתובת ה-API יכול לקרוא לה. פתרון עתידי: הוספת JWT Authentication.

**סטטוס:** ✅ מוגן ברמת ה-MVC, ⚠️ לא מוגן ברמת ה-API

### איום 4: Sensitive Data Exposure (חשיפת נתונים רגישים)

**רמת סיכון:** בינונית

**תיאור:** סיסמאות משתמשים נשמרות כ-Plaintext (טקסט גלוי) במסד הנתונים. אם מסד הנתונים דולף, כל הסיסמאות נחשפות.

**מצב נוכחי:**
```sql
-- ב-DB: Password = "1234" (plaintext!)
```

**פתרון עתידי מוצע:**
```csharp
// שימוש ב-BCrypt:
using BCrypt.Net;
string hashed = BCrypt.HashPassword("1234"); // "$2b$12$..."
bool isValid = BCrypt.Verify("1234", hashed); // true
// גם אם ה-DB דולף, Hash לא הפיך בפרקטיקה
```

**סטטוס:** ⚠️ חולשה ידועה – שיפור מתוכנן לגרסה הבאה

### איום 5: File Upload Vulnerability (חולשת העלאת קבצים)

**רמת סיכון:** בינונית

**תיאור:** תוקף עלול להעלות קובץ זדוני (למשל: `.php` שמריץ קוד על השרת) בתחפושת PDF.

**אמצעי הגנה שמומש:**
```csharp
// בדיקה 1: בדיקת סיומת בלבד (Client-Side)
var extension = Path.GetExtension(resumeFile.FileName).ToLower();
if (extension != ".pdf") { TempData["Error"] = "Only PDF files are allowed."; }

// בדיקה 2: בדיקת גודל (Server-Side)
if (resumeFile.Length > 5 * 1024 * 1024) { TempData["Error"] = "Max 5MB."; }

// בדיקה 3: שמירה בתיקיית wwwroot (ניתן לגישה ציבורית אך לא ניתן לביצוע)
// קבצים ב-wwwroot/uploads/ מוגשים כקבצים סטטיים – הם לא "מורצים" על ידי IIS/Kestrel
```

**מה שנשאר לשפר:** בדיקת **MIME Type** של הקובץ (לא רק הסיומת) והגבלת שמות תיקיות (Directory Traversal).

**סטטוס:** ✅ מוגן בסיסית, ⚠️ שיפור מומלץ

# פרק 15: מילון מונחים (Glossary)

## A

**API (Application Programming Interface):** ממשק תכנות יישומים. קבוצת כללים וחוזים שמגדירה כיצד רכיבי תוכנה מתקשרים זה עם זה. בפרויקט זה – REST API המאפשר ל-JobWebApp לתקשר עם JobWebService.

**async/await:** מתודולוגיית תכנות א-סינכרוני ב-C#. `async` מסמן שפונקציה עשויה לבצע פעולות שלוקחות זמן, ו-`await` מאפשר לה להמשיך לשרת בקשות אחרות בזמן ההמתנה.

**Authentication (אימות):** תהליך אימות זהות המשתמש (בדיקה שהוא מי שהוא טוען שהוא). בפרויקט: Email + Password.

**Authorization (הרשאה):** תהליך בדיקת הרשאות גישה (בדיקה שהמשתמש רשאי לבצע פעולה מסוימת). בפרויקט: SessionHelper.IsEmployee/IsEmployer/IsAdmin.

## B

**Backend:** החלק האחורי של המערכת, הרץ על השרת, אחראי על לוגיקה עסקית וגישה לנתונים. בפרויקט: JobWebService.

**BOM (Byte Order Mark):** שלושה bytes מיוחדים (EF BB BF) בתחילת קובץ UTF-8 המסמנים את קידוד הקובץ לתוכניות שקוראות אותו.

**Bridge Table:** טבלת גשר בבסיס נתונים המיישמת קשר Many-to-Many. בפרויקט: `JobApplication` מגשרת בין `User` ל-`Job`.

## C

**C# (C Sharp):** שפת תכנות מונחית-עצמים מבית Microsoft, הנמצאת בשימוש בכל שכבות הפרויקט.

**Client-Server:** ארכיטקטורת תוכנה שבה לקוח שולח בקשות לשרת, והשרת מעבדן ומחזיר תשובות.

**Connection String:** מחרוזת הגדרה המכילה את כל הפרמטרים לפתיחת חיבור למסד נתונים: ספק, שם קובץ, אישורים וכד'.

**Cookie:** קובץ טקסט קטן שהשרת שולח לדפדפן, והדפדפן מחזיר אותו בכל בקשה. משמש לניהול Sessions.

**CRUD:** ראשי תיבות של Create, Read, Update, Delete – ארבע הפעולות הבסיסיות על נתונים.

## D

**Data Access Layer:** שכבת הגישה לנתונים. אחראית על תקשורת עם מסד הנתונים בלבד.

**DBHelperOledb:** מחלקה מרכזית בפרויקט העוטפת `OleDbCommand` ו-`OleDbConnection` ומספקת ממשק פשוט לביצוע שאילתות SQL.

**Deserialization:** המרת מחרוזת JSON לאובייקט C#. מבוצעת על ידי `JsonSerializer.Deserialize<T>()`.

**DFD (Data Flow Diagram):** תרשים זרימת נתונים. מתאר כיצד מידע זורם בתוך המערכת ובין הישויות.

**DISTINCT:** מילת מפתח SQL שמבטיחה שכל ערך מוחזר פעם אחת בלבד (ללא כפילויות).

## E

**Employee:** בהקשר הפרויקט – מחפש עבודה. משתמש עם `UserTypeID = 2`.

**Employer:** בהקשר הפרויקט – מעסיק. משתמש עם `UserTypeID = 3`.

**Endpoint:** נקודת קצה – כתובת URL ספציפית ב-API המייצגת פונקציה מסוימת.

**Entity:** ישות – אובייקט מעולם הדומיין שמיוצג כטבלה במסד הנתונים.

**ERD (Entity Relationship Diagram):** תרשים קשרי ישויות. מתאר את הטבלאות ואת הקשרים ביניהן.

## F

**Foreign Key:** מפתח זר. שדה בטבלה שמצביע למפתח ראשי של טבלה אחרת, ומגדיר קשר בין שתי הטבלאות.

**Frontend:** החלק הקדמי של המערכת, הרץ בדפדפן, אחראי על תצוגה ואינטראקציה עם המשתמש. בפרויקט: JobWebApp (Razor Views).

**FOREIGN KEY:** אילוץ (Constraint) במסד נתונים שמבטיח שכל ערך בשדה מפתח זר קיים כמפתח ראשי בטבלה המפנה.

## G

**Genre:** תחום משרה. טבלת עזר המכילה קטגוריות (IT, Marketing, Finance וכד').

**GUID (Globally Unique Identifier):** מספר ייחודי של 128 bit, מיוצג כ-32 תווים הקסדצימליים. משמש בפרויקט לשמות קבצים ייחודיים ול-NotificationID.

## H

**HTTP (HyperText Transfer Protocol):** פרוטוקול תקשורת האינטרנט. מגדיר כיצד לקוחות ושרתים מתקשרים.

**HttpClient:** מחלקת .NET לביצוע בקשות HTTP. בפרויקט מנוהל כ-Singleton.

**HTML Encoding:** המרת תווים מיוחדים ל-HTML entities: `<` → `&lt;`, `>` → `&gt;`. מגן מפני XSS.

## I

**IDataReader:** ממשק C# לקריאת נתונים ממסד נתונים שורה-שורה. מוחזר על ידי `ExecuteReader()`.

**IRepository<T>:** ממשק גנרי המגדיר פעולות CRUD. מבטיח שכל Repository יממש Insert, Read, ReadAll, Update, Delete.

**Injection:** התקפת אבטחה שבה תוקף "מזריק" קוד זדוני לתוך בקשה לגיטימית.

## J

**JSON (JavaScript Object Notation):** פורמט טקסטואלי להחלפת נתונים. קריא לאדם, ניתן לעיבוד על ידי כל שפת תכנות. פורמט התקשורת בין JobWebApp ל-JobWebService.

**JobApplication:** טבלת קישור (Bridge Table) המייצגת הגשת מועמדות. כל רשומה מקשרת Employee ספציפי ל-Job ספציפי.

## L

**Lazy Initialization:** טכניקת אתחול עצלן – יצירת אובייקט רק בעת השימוש הראשון בו (ולא מראש). מיושמת ב-`LibraryUOW` עבור כל Repository.

**LEFT JOIN:** סוג JOIN ב-SQL שמחזיר את כל השורות מהטבלה השמאלית, גם אם אין התאמה בטבלה הימנית.

**LibraryUOW:** מחלקת ה-Unit of Work בפרויקט. מרכזת את כל ה-Repositories תחת גג אחד ומבטיחה שיתוף של חיבור ה-DB.

## M

**Many-to-Many:** קשר שבו כל רשומה בטבלה A יכולה להיות קשורה למספר רשומות בטבלה B ולהפך. מיושם בעזרת טבלת גשר.

**Microsoft Access (.accdb):** מערכת ניהול בסיסי נתונים של Microsoft. מיושמת כקובץ בודד על הדיסק. מתאימה לאפליקציות קטנות-בינוניות.

**MVC (Model-View-Controller):** תבנית ארכיטקטורה שמפרידה את האפליקציה לשלוש שכבות: Model (נתונים), View (תצוגה), Controller (לוגיקת בקרה).

**Multipart Form Data:** סוג Content-Type של HTTP שמאפשר שליחת נתונים מרובים כולל קבצים בינאריים בבקשה אחת.

## N

**.NET 8:** גרסת המסגרת של Microsoft לפיתוח תוכנה. LTS (Long Term Support) – תמיכה עד נובמבר 2026.

**NuGet:** מנהל חבילות עבור .NET. מאפשר הוסיף ספריות חיצוניות לפרויקט.

**Nullable (?):** אופרטור C# שמאפשר לסוג ערך (int, bool) לקבל ערך null. `int?` מאפשר null, `int` לא.

## O

**OleDb (Object Linking and Embedding Database):** ממשק API של Microsoft לגישה למסדי נתונים מסוגים שונים (Access, Excel, SQL Server ועוד).

**ORM (Object-Relational Mapping):** שכבה שממפה בין אובייקטים בשפת התכנות לטבלאות במסד נתונים. בפרויקט: שכבת ORM ביתית (לא Entity Framework).

## P

**Parameterized Query:** שאילתת SQL שבה ערכי הקלט מועברים כ-Parameters נפרדים (לא כחלק ממחרוזת ה-SQL). מגינה מפני SQL Injection.

**PDF (Portable Document Format):** פורמט קובץ סטנדרטי להצגת מסמכים. משמש בפרויקט לקובצי קורות חיים.

**Primary Key:** מפתח ראשי. שדה (או קבוצת שדות) שמזהה כל רשומה בטבלה באופן ייחודי.

## R

**Razor:** מנוע תבניות (Template Engine) של Microsoft שמשלב C# עם HTML. קבצי Razor הם `.cshtml`.

**Repository Pattern:** תבנית עיצוב שמבדדת את הלוגיקה של גישה לנתונים. כל Repository אחראי על CRUD של ישות אחת.

**REST (Representational State Transfer):** סגנון ארכיטקטורי לשירותי אינטרנט שמשתמש ב-HTTP Methods (GET/POST/PUT/DELETE) לפעולות שונות.

## S

**Serialization:** המרת אובייקט C# למחרוזת JSON. מבוצע על ידי `JsonSerializer.Serialize()`.

**Session:** מנגנון לאחסון נתוני משתמש בין בקשות HTTP. ב-ASP.NET Core: מאוחסן בשרת, מזוהה על ידי Cookie.

**Singleton:** תבנית עיצוב שמבטיחה שמחלקה מסוימת תיצור מופע אחד בלבד לכל אורך חיי האפליקציה.

**SQL (Structured Query Language):** שפת שאילתות למסדי נתונים יחסיים. כולל: SELECT, INSERT, UPDATE, DELETE, CREATE TABLE.

**SQL Injection:** התקפת אבטחה שמזריקה קוד SQL זדוני לתוך שאילתה לגיטימית.

## T

**Thread Pool:** מאגר חוטי ביצוע (Threads) שמנוהל על ידי .NET. `async/await` מאפשר שימוש יעיל בחוטים אלה.

**Transaction:** יחידת עבודה אטומית שמבטיחה שסדרת פעולות מסד נתונים מתבצעת כולה או לא מתבצעת כלל.

## U

**Unit of Work (UoW):** תבנית עיצוב שמרכזת מספר פעולות מסד נתונים תחת Transaction אחד. מיושמת ב-`LibraryUOW`.

**UNION:** פקודת SQL שמאחדת תוצאות של שתי שאילתות SELECT או יותר.

**UTF-8:** קידוד תווים אוניברסלי שתומך בכל שפות העולם (כולל עברית) ומייצג כל תו ב-1-4 bytes.

## V

**ViewModel:** מחלקת C# שמכילה את הנתונים הנחוצים לתצוגת מסך ספציפי (אוסף של מודלים). בפרויקט: `HomePage`, `JobCatalog`, `JobHistory`.

**ViewBag:** מילון דינמי ב-ASP.NET Core MVC שמאפשר העברת נתונים מ-Controller ל-View.

## W

**Web API:** שירות אינטרנט שחושף פונקציונליות דרך HTTP Endpoints. בפרויקט: JobWebService.

**wwwroot:** תיקיית ברירת המחדל ב-ASP.NET Core לקבצים סטטיים (HTML, CSS, JS, תמונות). בפרויקט: כוללת גם את תיקיית `uploads/` לקבצי PDF.

## X

**XSS (Cross-Site Scripting):** התקפת אבטחה שמזריקה JavaScript זדוני לדפים שמוצגים למשתמשים אחרים.

## 16.2 יתרונות ייחודיים של JobFindUltimate

1. **פשטות הגדרה:** ניתן להפעיל את המערכת בכ-5 דקות על כל מחשב Windows עם .NET 8.
2. **קוד פתוח:** כל שורת קוד גלויה, ניתן לבדיקה ולהרחבה.
3. **אין תלות בשירות חיצוני:** המערכת פועלת Offline לחלוטין – אין צורך בחיבור לענן.
4. **Customizable מלא:** כל ישות, כל Endpoint, כל דף UI ניתן לשינוי מלא.
5. **ערכית לימודית:** הקוד מתועד ברמה גבוהה ומתאים ללמידה.

## 16.3 חסרונות לעומת פתרונות מסחריים

1. **אבטחה:** אין JWT, אין HTTPS מוגדר, אין Rate Limiting.
2. **Scale:** Access אינו מתאים לעשרות אלפי משתמשים בו-זמנית.
3. **תכונות AI:** אין ML Matching, אין חיפוש סמנטי.
4. **אינטגרציות:** אין LinkedIn Import, אין Email Notifications, אין Calendar.
5. **Mobile App:** אין אפליקציה ייעודית ל-iOS/Android.
