# פרק 1: מבוא

## 1.1 שם הפרויקט ותיאורו הכללי

שם הפרויקט הוא **JobFindUltimate** – מערכת תוכנה מקיפה מבוססת אינטרנט (Web-Based System) המשמשת כפלטפורמה מרכזית לחיפוש עבודה, פרסום משרות, ניהול תהליכי גיוס עובדים ותקשורת ישירה בין מועמדים למעסיקים. שם הפרויקט מורכב משלוש מילים: "Job" (עבודה), "Find" (מצא) ו-"Ultimate" (מוחלט/סופי), ומייצג את שאיפת הפרויקט להיות הפלטפורמה הטובה ביותר, המקיפה ביותר וה"סופית" עבור כל צורכי חיפוש העבודה.

המערכת מממשת ארכיטקטורה מודרנית של **Client-Server** המפרידה לחלוטין בין שכבת העיבוד (Backend) לבין שכבת התצוגה (Frontend). שרת ה-Backend מממש **ASP.NET Core Web API** ומשמש כנקודת הגישה המרכזית לכל הנתונים, בעוד שאפליקציית הלקוח (Frontend) מממשת **ASP.NET Core MVC** ומבצעת קריאות HTTP א-סינכרוניות לשרת ה-API. הנתונים עצמם נשמרים במסד נתונים יחסי מסוג **Microsoft Access (.accdb)**, הנגיש על ידי שכבת ה-ORM הביתית שנכתבה עבור הפרויקט.

מבחינה פונקציונלית, המערכת מאפשרת:
- לאורח (Guest) לגלוש בקטלוג משרות, לסנן תוצאות לפי מגוון קריטריונים ולהירשם למערכת.
- למחפש עבודה (Employee) לחפש משרות, לסנן אותן, להגיש מועמדות בצירוף קורות חיים בפורמט PDF, לעקוב אחר היסטוריית המועמדויות שלו ולנהל צ'אט ישיר עם מעסיקים.
- למעסיק (Employer) לפרסם משרות חדשות עם פרמטרים מפורטים, לנהל את המשרות שלו, לצפות במועמדים, לסנן אותם לפי תוכן קורות החיים, ולנהל את עובדיו (עדכון שכר, הפסקת העסקה).
- למנהל המערכת (Admin) לפקח על כלל תוכן המערכת, למחוק משרות ומשתמשים בעייתיים, לסקור מועמדויות ולשלוח הודעות מערכת גלובליות לכלל המשתמשים.

## 1.2 קהל היעד

מערכת **JobFindUltimate** תוכננה לשרת שלוש קבוצות משתמשים עיקריות, כשלכל אחת מהן ממשק ייעודי ומותאם לצרכיה הספציפיים:

### קבוצה ראשונה: מחפשי עבודה (Employees)

קבוצה זו כוללת אנשים בכל שלב בחיים המחפשים הזדמנות תעסוקתית חדשה. הם יכולים להיות בוגרי תיכון המחפשים עבודה ראשונה, אנשי מקצוע ותיקים המחפשים שינוי קריירה, או אנשים שיצאו לפנסיה מוקדמת ומחפשים עבודה חלקית. צרכיהם המרכזיים הם: גישה מהירה ונוחה לקטלוג עשיר של משרות, יכולת סינון ומיון עדכנית ואמינה, אפשרות להגיש מועמדות בפשטות ללא תהליכים בירוקרטיים מסורבלים, ומעקב שקוף אחר סטטוס כל מועמדות שהגישו.

### קבוצה שנייה: מעסיקים (Employers)

קבוצה זו כוללת חברות של כל הגדלים, עסקים פרטיים, ארגונים ממשלתיים ועמותות המעוניינים לגייס עובדים. מנהלי גיוס (HR Managers), בעלי עסקים ומנהלים אופרטיביים שייכים לקבוצה זו. צרכיהם הם: יכולת לפרסם משרה מפורטת בפשטות, פרסום לקהל יעד רחב ומדויק ככל האפשר, כלים יעילים לסינון ומיון מועמדים, תקשורת ישירה ומהירה עם מועמדים מתאימים, וניהול כל תהליך הגיוס ממקום אחד.

### קבוצה שלישית: מנהלי מערכת (Admins)

קבוצה זו כוללת את הגורמים האחראים על תפעול שוטף של הפלטפורמה. תפקידם הוא לפקח על איכות התוכן, למנוע פרסום משרות מזויפות או פוגעניות, לטפל בתלונות משתמשים, ולשמור על בריאות המערכת ושלומה הטכני.

## 1.3 הסיבות לבחירת הנושא

בחירת הנושא "מערכת לחיפוש עבודה" נבעה ממספר שיקולים משמעותיים:

**שיקול ראשון – רלוונטיות חברתית:** שוק העבודה הוא אחד התחומים המשמעותיים ביותר בחיי האדם המודרני. היכולת לאתר הזדמנות תעסוקתית מתאימה במהירות ובדייקנות משפיעה ישירות על רווחתם הכלכלית והנפשית של אנשים. פרויקט שפותר בעיה ממשית ואוניברסלית שכזו בעל ערך חברתי אמיתי, ולא רק ערך אקדמי.

**שיקול שני – מורכבות טכנית מיטבית:** פרויקט מסוג זה דורש מימוש של כמעט כל היבטי חלופת שירותי הרשת: REST API עם CRUD מלא, ארכיטקטורת Client-Server, קריאות HTTP א-סינכרוניות, ניהול Sessions, העלאת קבצים (File Upload), מסד נתונים יחסי עם מספר טבלאות ויחסי Many-to-Many, הרשאות גישה מבוססות תפקיד (Role-Based Authorization) ומערכת הודעות בין משתמשים (Chat System). כל אלה מייצגים אתגרים טכניים אמיתיים שמאפשרים למידה עמוקה.

**שיקול שלישי – חשיפה לטכנולוגיות עכשוויות:** ביצוע הפרויקט חשף אותי לטכנולוגיות ולעקרונות עיצוב הנמצאים בשימוש נרחב בתעשיית התוכנה בישראל ובעולם, כגון ASP.NET Core, REST APIs, JSON, ו-HttpClient אסינכרוני.

**שיקול רביעי – אפשרות לייצוגיות אישית:** הפרויקט מאפשר יצירה של מוצר דיגיטלי אמיתי, שאם ירצה, התלמיד יכול להציג בפני מעסיקים פוטנציאליים כחלק מתיק העבודות שלו לקראת כניסה לתעשיית ההיי-טק.

## 1.4 מטרות המערכת

### מטרות על (Primary Goals)

**מטרה א' – גישור דיגיטלי בין מחפשי עבודה למעסיקים:**
יצירת פלטפורמה אחידה שבה מחפש עבודה ומעסיק יכולים להיפגש בחלל וירטואלי ולבצע את כל שלבי התהליך הראשוני של הגיוס – מפרסום ואיתור המשרה, דרך הגשת המועמדות, ועד לתקשורת ישירה – מבלי לדרוש תיווך חיצוני.

**מטרה ב' – ניהול מחזור חיים מלא של מועמדות:**
ניהול מלא של מחזור החיים (Lifecycle) של כל מועמדות: מרגע הגשתה (Status: "Submitted"), דרך שלב הסינון, ועד לתוצאה הסופית (Status: "Accepted" / "Rejected"). כל שלב מתועד במסד הנתונים ונגיש למחפש העבודה ולמעסיק כאחד.

**מטרה ג' – הנגשת מידע מיידי ועדכני:**
המערכת נועדה להציג תמיד מידע עדכני הישירות ממסד הנתונים. בניגוד לפתרונות סטטיים של פרסום קבצי Excel מייצאים, כל שינוי שמבצע מעסיק (פרסום משרה חדשה, שינוי סטטוס משרה) מיד גלוי לכל המשתמשים ללא דיליי.

### מטרות נלוות (Secondary Goals)

**מטרה נלווית א' – שיפור חוויית המועמד באמצעות קורות חיים מקוונים:**
הפרויקט מיישם מנגנון שבו מחפש העבודה יכול לבנות ולשמור את קורות החיים שלו ישירות בפלטפורמה, ולאחר מכן להגיש אותם בלחיצת כפתור בודדת לכל משרה שרלוונטית לו. בנוסף, ניתן לצרף קורות חיים בפורמט PDF ישירות בעת הגשת המועמדות.

**מטרה נלווית ב' – מתן כלי ניהול עובדים בסיסי למעסיקים:**
המערכת מספקת למעסיק לא רק כלי לגיוס, אלא גם כלי ניהולי בסיסי: עדכון שכר עובדים שכבר גויסו, וסימון עובדים שסיימו העסקתם.

**מטרה נלווית ג' – מנגנון התראות מערכת:**
מנהל המערכת יכול לשלוח הודעות מערכת גלובליות לכל המשתמשים, המאפשרות לו לעדכן אותם על שינויים, תחזוקה מתוכננת או עדכונים חשובים בפלטפורמה.

**מטרה נלווית ד' – מערכת ביקורות (Reviews):**
המערכת כוללת תשתית למנגנון ביקורות שבו עובדים יכולים לדרג ולהעריך מעסיקים שעבדו עבורם, ובכך לסייע למחפשי עבודה עתידיים לקבל החלטות מושכלות יותר.

## 1.5 תיאור כללי של המערכת

**JobFindUltimate** היא מערכת תוכנה מבוזרת (Distributed System) המורכבת משני יישומים עצמאיים הפועלים במקביל:

**יישום 1 – JobWebService (שרת ה-API):**
יישום ASP.NET Core Web API הפועל כשרת בלתי-תלוי (Standalone Server) ומספק ממשק REST API. השרת מקבל בקשות HTTP מהלקוח, מעבד אותן, מבצע פעולות קריאה/כתיבה במסד הנתונים Microsoft Access באמצעות ספריית OleDb, ומחזיר תשובות בפורמט JSON.

**יישום 2 – JobWebApp (לקוח ה-MVC):**
יישום ASP.NET Core MVC הפועל כשרת-אינטרנט (Web Server) לצד הלקוח. הוא מקבל בקשות HTTP מהדפדפן של המשתמש, מבצע קריאות API לשרת ה-Web Service, מרכיב את הנתונים שהתקבלו לתוך תבניות HTML (Razor Views) ומחזיר את הדף המוכן לדפדפן.

**בסיס הנתונים – JobFindUltimate.accdb:**
קובץ Microsoft Access המכיל את כל הנתונים הפרסיסטנטיים של המערכת: טבלת משתמשים, טבלת משרות, טבלת מועמדויות, טבלת ארצות, טבלת תחומי משרה, טבלת ביקורות, טבלת התראות וטבלת הודעות צ'אט.

**ספריית מודלים משותפת – JobModels:**
ספריית Class Library של C# המכילה את כל מחלקות ה-Model ואת ה-ViewModels. ספרייה זו מוזכרת (Referenced) הן על ידי פרויקט השרת והן על ידי פרויקט הלקוח, וכך מובטחת עקביות מוחלטת בין המבנה שהשרת שולח לבין המבנה שהלקוח מצפה לקבל.

## 1.6 גבולות המערכת

הגדרת גבולות המערכת חיונית להבנה ברורה של מה המערכת כן מבצעת ומה היא אינה מבצעת.

### מה המערכת מבצעת (In Scope):
- רישום, אימות והתחברות משתמשים על פי שלושה תפקידים: מחפש עבודה, מעסיק ומנהל מערכת.
- פרסום משרות עם פרמטרים מפורטים (כותרת, תיאור, סוג, פילטר, ארץ, תחום, מעסיק).
- חיפוש וסינון משרות לפי קריטריונים מרובים.
- הגשת מועמדות למשרות, כולל העלאת קורות חיים בפורמט PDF.
- ניהול שלב המועמדות (Accepted/Rejected/Submitted).
- מערכת הודעות פנימית (Chat) בין שני משתמשים.

## 1.8 ארכיטקטורת שכבות המערכת

המערכת בנויה בארכיטקטורת **N-Tier Architecture** (ארכיטקטורת שכבות) המפרידה את האחריות לשכבות עצמאיות ומבודדות. ספציפית, בפרויקט זה מדובר בארכיטקטורת **Three-Tier** (שלוש שכבות מרכזיות):

### שכבה 1 – שכבת התצוגה (Presentation Layer)
מיושמת על ידי פרויקט **JobWebApp** ותבניות ה-Razor Views. שכבה זו אחראית אך ורק על הצגת הנתונים למשתמש קצה ועל קבלת הקלט ממנו. היא אינה מכילה שום לוגיקה עסקית ואינה פונה ישירות למסד הנתונים. כל אינטראקציה עם הנתונים מתבצעת דרך בקשות HTTP לשרת ה-API.

### שכבה 2 – שכבת הלוגיקה העסקית (Business Logic Layer)
מיושמת על ידי בקרי ה-API (**Controllers**) בפרויקט **JobWebService**. שכבה זו אחראית על הלוגיקה של המערכת: בדיקת תקינות קלט, אכיפת כללי עסק (Business Rules) כגון "לא ניתן להגיש מועמדות למשרה סגורה" ו"שם משתמש חייב להיות ייחודי", ותיאום בין הבקשה הנכנסת לבין שכבת הנתונים.

### שכבה 3 – שכבת הנתונים (Data Access Layer)
מיושמת על ידי מחלקת **DBHelperOledb**, מחלקות ה-**Repository** (אחת לכל ישות), ויחידת העבודה **LibraryUOW**. שכבה זו אחראית על כל האינטראקציה עם מסד הנתונים: בניית שאילתות SQL, הוספת פרמטרים מאובטחים, ביצוע הפעולה ומיפוי התוצאות חזרה לאובייקטי C#.

```
[ דפדפן המשתמש ]
        │  HTTP Request (GET/POST)
        ▼
[ JobWebApp MVC Controllers ]  ◄── Session / Cookie Management
        │  async HTTP → JSON
        ▼
[ JobWebService API Controllers ]  ◄── Business Rules Validation
        │
        ▼
[ LibraryUOW + Repositories ]  ◄── Repository Pattern
        │
        ▼
[ DBHelperOledb ]  ◄── OleDbCommand + Parameterized Queries
        │
        ▼
[ Microsoft Access .accdb ]
```

## 1.10 אתגרים מרכזיים ופתרונות

### אתגר 1 – נעילת בסיס נתונים (Database Locking) ב-MS Access

**תיאור הבעיה:** Microsoft Access אינו מסד נתונים ייעודי לסביבות שרת (Server-Side). מנגנון הנעילה שלו (Locking Mechanism) פשוט ביחס למסדי נתונים ייעודיים כגון SQL Server. כאשר מספר בקשות HTTP מגיעות בו-זמנית לשרת, ושתיהן מנסות לפתוח חיבור OleDb בו-זמנית, עלולות להיווצר שגיאות מסוג "Could not use file, already in use" או נעילות מגומלין (Deadlocks).

**הפתרון שמומש:** הפתרון מורכב משני חלקים:
1. **ניהול מחזור חיים קפדני של החיבור:** בכל Method בבקרי ה-API, החיבור נפתח (`OpenConnection`) רק ממש לפני ביצוע הפעולה, ונסגר (`CloseConnection`) מיד בסיומה בבלוק `finally`. הבלוק `finally` מבטיח שהחיבור ייסגר גם במקרה של חריגה (Exception), ובכך מונע "דלף" (Leak) של חיבורים פתוחים.

```csharp
// דפוס שחוזר בכל Method של בקר ה-API:
try
{
    this.libraryUOW.HelperOledb.OpenConnection();
    // ... ביצוע הפעולה
}
catch (Exception ex)
{
    Trace.WriteLine(ex);
    return false; // או ערך ברירת מחדל
}
finally
{
    this.libraryUOW.HelperOledb.CloseConnection(); // תמיד מתבצע
}
```

2. **מחלקת `DBHelperOledb` כ-Single Shared Object:** המחלקה מוגדרת ב-`Program.cs` של השרת כ-**Singleton** (`builder.Services.AddSingleton<DBHelperOledb>()`), מה שאומר שקיים אובייקט אחד ויחיד של מחלקה זו לכל אורך חיי היישום. אובייקט ה-`OleDbCommand` שבתוכה נוצר פעם אחת ומשמש שוב ושוב, תוך ניקוי פרמטרים (`Parameters.Clear()`) לאחר כל פעולה.

### אתגר 2 – מעבר לתכנות א-סינכרוני (Async/Await)

**תיאור הבעיה:** בתכנות מסורתי (סינכרוני), כל קריאת רשת חוסמת (Blocks) את ה-Thread הנוכחי עד לקבלת התשובה. אם שרת ה-MVC מקבל 50 בקשות בו-זמנית, ו-Thread Pool מוגבל ל-25 Threads, בקשות 26-50 ימתינו בתור (Queue) עד ש-Thread יתפנה. זה מוביל לעומסים גבוהים שבהם הממשק נראה קפוא ולא מגיב.

**הפתרון שמומש:** כל ה-Action Methods בבקרי ה-MVC (`EmployeeController`, `EmployerController`, `GuestController` ו-`AdminController` בצד הלקוח) מוגדרים כ-`async Task<IActionResult>`. קריאת ה-HTTP ל-API מבוצעת עם `await client.GetAsync()` / `await client.PostAsync()`. בעת ה-`await`, ה-Thread מוחזר ל-Thread Pool ועשוי לשרת בקשות אחרות. כאשר התשובה מהשרת מגיעה, ASP.NET Core שוקל Thread לביצוע המשך הקוד.

```csharp
// דוגמה מתוך EmployeeController.cs:
public async Task<IActionResult> Home()
{
    ApiClient<List<Job>> client = BuildClient<List<Job>>("Guest", "GetAllJobs");
    // await מאפשר ל-Thread לשרת בקשות אחרות בזמן ההמתנה
    List<Job> jobs = (await client.GetAsync() ?? new List<Job>())
        .Where(job => job.JobStatus != false)
        .ToList();
    // ...
}
```

### אתגר 3 – הגנה מפני SQL Injection

**תיאור הבעיה:** SQL Injection היא אחת מהתקפות האינטרנט הנפוצות והמסוכנות ביותר. תוקף יכול להזין במקום ערך שדה תקין מחרוזת כמו: `' OR '1'='1' --`, שאם משורשרת ישירות לתוך שאילתת SQL, משנה את לוגיקת השאילתה לחלוטין ועלולה לאפשר גישה לכל הנתונים ואפילו מחיקתם.

**הפתרון שמומש:** כל שאילתות ה-SQL בפרויקט עוברות דרך מנגנון הפרמטריזציה של `OleDbCommand`. במקום שרשור מחרוזות, הפרמטרים מוסיפים בנפרד ומנוקים לפי כלל OleDb. הנה הדפוס המדויק:

```csharp
// במקום מסוכן: "SELECT * FROM [User] WHERE UserName='" + userName + "'"
// הפתרון המאובטח:
public bool IsAvailableUserName(string userName)
{
    string sql = "SELECT 1 FROM [User] WHERE UserName=@UserName";
    this.helperOleDb.AddParameters("@UserName", userName); // הפרמטר נרשם בנפרד
    using (IDataReader dr = this.helperOleDb.Read(sql))
    {
        return !dr.Read();
    }
}
// AddParameters מיישמת:
public void AddParameters(string name, object value)
{
    this.oleDbCommand.Parameters.Add(new OleDbParameter(name, value ?? DBNull.Value));
}
```

### אתגר 4 – ניהול סוגי משתמשים ומסלולי ניתוב

**תיאור הבעיה:** המערכת כוללת ארבעה סוגי משתמשים (Guest, Employee, Employer, Admin), כשלכל סוג יש ממשק שונה, הרשאות שונות ומסכים שונים. כיצד מבטיחים שמשתמש מסוג Employer לא יוכל לגשת לעמוד Employee/Resume ולעדכן את קורות החיים של מישהו אחר?

**הפתרון שמומש:** מחלקת `SessionHelper` המאחסנת את `UserTypeID` ב-Session מוצפן ב-Cookie. בתחילת כל Action Method בבקרי הלקוח, מתבצעת בדיקת הרשאה:

```csharp
private bool IsAuthorized()
{
    return SessionHelper.IsEmployee(HttpContext.Session);
    // IsEmployee בודקת: GetUserTypeID(session) == 2
}
// בכל Action Method:
public async Task<IActionResult> Resume()
{
    if (!IsAuthorized()) return RedirectToAction("Home", "Guest");
    // ...
}
```

### אתגר 5 – מצב עוגיה של MS Access עם JOIN מורכב

**תיאור הבעיה:** Microsoft Access משתמש בתחביר SQL קל שונה מ-SQL Server. בפרט, JOIN עם שלוש טבלאות ויותר דורש בריכוז (Nesting) של הסוגריים.

**הפתרון שמומש:** השאילתה בORM מנוסחת בצורה המדויקת ל-Access:

```csharp
// שאילתה מורכבת עם שתי LEFT JOINs בתחביר Access:
string sql = "SELECT Job.*, Country.CountryName, Genre.GenreTitle " +
             "FROM (Job LEFT JOIN Country ON Job.CountryID = Country.CountryID) " +
             "LEFT JOIN Genre ON Job.GenreID = Genre.GenreID";
```

- **ריכוזיות:** כל הפעולות מתבצעות תחת פלטפורמה אחת.
- **שקיפות:** מחפש העבודה רואה תמיד את סטטוס מועמדותו.
- **מהירות:** הגשת מועמדות אורכת פחות מדקה.
- **שיתוף מידע:** ביקורות של עובדים על מעסיקים עוזרות לקבל החלטות מושכלות.

## 2.4 מצב עתידי

**צד מחפש העבודה:**

5. הוא רואה בזמן אמת את סטטוס מועמדותו בדף "Job History".
6. הוא מנהל שיחה ישירה עם המעסיק דרך מערכת הצ'אט.

**צד המעסיק:**

## 2.5 ERD – דיאגרמת קשרי ישויות

דיאגרמת קשרי הישויות (Entity Relationship Diagram) מתארת את המבנה הלוגי של בסיס הנתונים ואת הקשרים בין הישויות השונות.

### ישויות המערכת וקשרים ביניהן

| ישות | פירוש | מספר רשומות טיפוסי |
|---|---|---|
| Users | כל המשתמשים של המערכת | מאות |
| Jobs | כל המשרות שפורסמו | אלפים |
| JobApplication | הגשות מועמדות | עשרות אלפים |
| Genre | תחומי משרה | עשרות |
| Country | מדינות | מאות |
| Reviews | ביקורות על מעסיקים | אלפים |
| Notification | הודעות מערכת | עשרות |
| ChatMessage | הודעות צ'אט | מאות אלפים |

### הקשרים המרכזיים

**קשר 1 – User ↔ UserType (Many-to-One):**
כל משתמש שייך לסוג משתמש אחד בדיוק. סוג המשתמש קובע את הרשאותיו וממשקו. יחס זה מיושם על ידי מפתח זר `UserTypeID` בטבלת `User`.

**קשר 2 – Job ↔ User/Employer (Many-to-One):**
כל משרה פורסמה על ידי מעסיק אחד בדיוק. מעסיק אחד יכול לפרסם משרות רבות. יחס זה מיושם על ידי מפתח זר `EmployerID` בטבלת `Job` המצביע ל-`UserID` בטבלת `User`.

**קשר 3 – Job ↔ JobApplication ↔ User/Employee (Many-to-Many):**
זהו הקשר המרכזי ביותר במערכת. מחפש עבודה אחד יכול להגיש מועמדות למשרות רבות. משרה אחת יכולה לקבל מועמדות מעובדים רבים. קשר Many-to-Many מיושם באמצעות טבלת קישור – `JobApplication` – המכילה מפתח זר `JobID` ומפתח זר `EmployeeID`.

**קשר 4 – Job ↔ Genre (Many-to-One):**
כל משרה שייכת לתחום (Genre) אחד בדיוק. תחום אחד יכול להכיל משרות רבות. יחס זה מאפשר את הסינון לפי קטגוריה.

**קשר 5 – Job ↔ Country (Many-to-One):**
כל משרה ממוקמת במדינה אחת. מדינה אחת יכולה לארח משרות רבות. יחס זה מאפשר סינון לפי מיקום גיאוגרפי.

**קשר 6 – User ↔ ChatMessage (Many-to-Many Self-Referencing):**
כל הודעת צ'אט קושרת שני משתמשים: שולח (SenderID) ומקבל (ReceiverID). שני המפתחות הזרים מצביעים לאותה טבלת `User`, וזהו קשר Self-Referencing.

## 2.7 דיאגרמות Use Case

### Use Case 1: אורח ומחפש עבודה

**שחקנים:** Guest (אורח), Employee (מחפש עבודה).

**תרחישים עיקריים:**

**UC-G1: גלישה בקטלוג משרות**
- *שחקן:* Guest / Employee
- *תנאי פתיחה:* המשתמש נמצא בדף הבית
- *זרימה ראשית:* משתמש לוחץ על "Job Catalog" בתפריט הניווט → המערכת שולפת את כל המשרות מ-API → מציגה אותן בתצוגת רשת (Grid) עם כרטיסי משרה.
- *זרימה חלופית:* אם אין משרות פעילות – מוצגת הודעה "No jobs found".
- *תנאי סיום:* קטלוג המשרות מוצג.

**UC-G2: סינון משרות**
- *שחקן:* Guest / Employee
- *תנאי פתיחה:* UC-G1 הושלם
- *זרימה ראשית:* משתמש בוחר ערכים בסרגל הסינון (ארץ, תחום, סוג משרה, חיפוש חופשי) → המערכת מסננת את המשרות בצד הלקוח (Client-Side Filtering) בזמן אמת → מוצגות רק המשרות המתאימות.
- *תנאי סיום:* רשימה מסוננת מוצגת.

**UC-G3: רישום למערכת**
- *שחקן:* Guest
- *תנאי פתיחה:* Guest לוחץ על כפתור "Register"
- *זרימה ראשית:* מוצג טופס רישום עם שדות: שם פרטי, שם משפחה, שם משתמש, דוא"ל, סיסמה, ארץ, טלפון, סוג משתמש (Employee/Employer).
- *ולידציות:* שם משתמש ייחודי (בדיקת API), דוא"ל ייחודי (בדיקת API), שדות חובה לא ריקים.
- *תנאי סיום מוצלח:* רשומת User נשמרת; משתמש מנותב לדף ההתחברות.
- *תנאי סיום כושל:* הודעת שגיאה מוצגת.

**UC-E1: הגשת מועמדות למשרה**
- *שחקן:* Employee (מחובר)
- *תנאי פתיחה:* Employee מסתכל על כרטיס משרה בקטלוג
- *זרימה ראשית:* Employee לוחץ "Apply" → מוצגת חלונית מודאל (Modal) → Employee בוחר: להגיש עם קורות החיים המקוונים הקיימים, או לצרף קובץ PDF חדש → לוחץ "Submit".
- *ולידציות:* אם קובץ מצורף – חייב להיות PDF ועד 5MB.
- *זרימה טכנית:* קובץ נשמר ב-`wwwroot/uploads/` עם שם GUID ייחודי; ה-URL נשלח ל-API `User/ApplyToJob`; רשומת `JobApplication` נוצרת.
- *תנאי סיום מוצלח:* הודעת "Application submitted successfully!" מוצגת.

**UC-E2: עדכון קורות חיים מקוונים**
- *שחקן:* Employee (מחובר)
- *תנאי פתיחה:* Employee נמצא בדף "Resume"
- *זרימה ראשית:* Employee מעלה קובץ PDF של קורות החיים → API `User/UpdateOnlineResume` מעדכן את `ResumeText` ב-User.
- *תנאי סיום:* קורות החיים שמורים ונגישים להגשה עתידית.

### Use Case 2: מעסיק (Employer)

**UC-EM1: פרסום משרה חדשה**
- *שחקן:* Employer (מחובר)
- *תנאי פתיחה:* Employer לוחץ "Post Job" בתפריט
- *זרימה ראשית:* Employer ממלא טופס עם: כותרת, תיאור, סוג משרה, ארץ, תחום (Genre), JobFilter (פרטים נוספים), סטטוס ראשוני (פעיל) → לוחץ "Post".
- *ולידציות שרת:* בדיקה שכל שדות החובה (JobTitle, JobDescription, JobType, EmployerID, CountryID, GenreID) אינם ריקים.
- *זרימה טכנית:* API `Employer/AddJob` קורא ל-`JobRepository.Create(job)` → INSERT לטבלת Job.
- *תנאי סיום:* משרה פורסמה; גלויה בקטלוג.

**UC-EM2: צפייה במועמדים וסינונם**
- *שחקן:* Employer (מחובר)
- *זרימה ראשית:* Employer בוחר משרה מרשימת המשרות שלו → המערכת שולפת ממשק "Applications" עם רשימת כל מועמדי אותה משרה → Employer מזין מילת מפתח לחיפוש בקורות חיים → הרשימה מסתננת.
- *זרימה טכנית:* API `Employer/Applications?jobId=X&resumeWord=Y` → `JobApplicationRepository.ReadByJobId(jobId)` → סינון Client-Side לפי `ResumeSnapshot.Contains(resumeWord)`.

**UC-EM3: שינוי סטטוס משרה (Toggle)**
- *שחקן:* Employer
- *זרימה ראשית:* Employer לוחץ "Toggle Status" על משרה ברשימה → API `Employer/ToggleJobStatus?jobId=X` שולף המשרה, הופך את `JobStatus` (`true → false` או `false → true`), ומעדכן.

### Use Case 3: מנהל מערכת (Admin)

**UC-A1: מחיקת משרה**
- *שחקן:* Admin (מחובר, UserTypeID = 4)
- *זרימה:* Admin רואה בתצוגת "Admin/Home" את כל המשרות → בוחר משרה ולוחץ "Delete" → API `Admin/DeleteJob` → `JobRepository.Delete(jobId)`.

**UC-A2: מחיקת משתמש**
- *שחקן:* Admin
- *זרימה:* Admin גולש לרשימת המשתמשים → בוחר ולוחץ "Delete User" → API `Admin/DeleteUser` → `UserRepository.Delete(userId)`.

**UC-A3: שליחת הודעת מערכת**
- *שחקן:* Admin
- *זרימה:* Admin עובר לעמוד "Notify" → מזין טקסט הודעה → API `Admin/NotifyAboutWebsiteChanges` → יצירת `Notification` חדש עם `NotificationID = Guid.NewGuid().ToString("N")` ושמירה בטבלת `Notification`.

## 3.2 טבלאות ראשיות

### טבלה 1: `User`

טבלה זו היא אחת הטבלאות המרכזיות ביותר במערכת. היא מאחדת תחת גג אחד את כל סוגי המשתמשים: מחפשי עבודה, מעסיקים ומנהלי מערכת. ההפרדה ביניהם מתבצעת על ידי שדה `UserTypeID`.

| שם השדה | סוג נתונים | אילוץ | תיאור ותפקיד מפורט |
|---|---|---|---|
| `UserID` | VARCHAR(50) | PRIMARY KEY, NOT NULL | מזהה ייחודי של המשתמש. נוצר כמספר COUNTER ב-Access או כמחרוזת ייחודית. מפתח ראשי של הטבלה. |
| `FirstName` | VARCHAR(100) | NOT NULL | שם פרטי של המשתמש. משמש להצגה בממשק ("ברוך הבא, [שם פרטי]"). |
| `LastName` | VARCHAR(100) | NOT NULL | שם משפחה של המשתמש. |
| `UserName` | VARCHAR(100) | NOT NULL, UNIQUE | שם המשתמש המשמש לזיהוי בהתחברות. חייב להיות ייחודי במערכת. הייחודיות נאכפת ברמת הקוד על ידי `IsExistUserName()` לפני INSERT. |
| `Password` | VARCHAR(255) | NOT NULL | סיסמת המשתמש. בגרסה הנוכחית נשמרת כטקסט פשוט (Plaintext). שיפור עתידי: הצפנת MD5 או BCrypt. |
| `Country` | VARCHAR(50) | NULL | מספר זיהוי של הארץ שבה מתגורר המשתמש. מקושר ל-`Country.CountryID` (לא כ-FK פורמלי ב-Access). |
| `PhoneNum` | VARCHAR(20) | NULL | מספר טלפון ליצירת קשר. |
| `CreationDate` | VARCHAR(20) | NULL | תאריך יצירת החשבון בפורמט `yyyy-MM-dd`. |
| `UserTypeID` | INTEGER | FOREIGN KEY → UserType | קובע אם המשתמש הוא Guest(1)/Employee(2)/Employer(3)/Admin(4). |
| `Email` | VARCHAR(200) | NOT NULL, UNIQUE | כתובת הדוא"ל. משמשת גם כשדה התחברות חלופי ל-UserName. |
| `ResumeText` | MEMO | NULL | תוכן קורות החיים המקוונים, או URL לקובץ ה-PDF שהועלה. |
| `Salary` | DECIMAL | NULL | השכר הנוכחי/המבוקש. מעודכן על ידי המעסיק. |
| `CountryName` | Virtual/JOIN | - | שם הארץ (אינו שדה בטבלה! מתמלא על ידי JOIN עם טבלת Country בשאילתת SELECT). |

**שאילתת SELECT מייצגת:**
```sql
SELECT [User].*, Country.CountryName 
FROM [User] 
LEFT JOIN Country ON [User].Country = CStr(Country.CountryID)
WHERE [User].UserID=@UserID
```
שאילתה זו מבצעת LEFT JOIN עם טבלת Country ומוסיפה את שם הארץ לאובייקט ה-User המוחזר. פונקציית `CStr()` ב-Access ממירה את `CountryID` (מספרי) למחרוזת להשוואה עם שדה `Country` (טקסטואלי) בטבלת User.

### טבלה 2: `UserType`

טבלת עזר (Lookup Table) המגדירה את תפקידי המשתמשים.

| שם השדה | סוג נתונים | אילוץ | תיאור |
|---|---|---|---|
| `UserTypeID` | INTEGER | PRIMARY KEY | מזהה ייחודי של הסוג |
| `UserTypeName` | VARCHAR(50) | NOT NULL | שם הסוג: "Guest", "Employee", "Employer", "Admin" |

**ערכים טיפוסיים:**
| UserTypeID | UserTypeName |
|---|---|
| 1 | Guest |
| 2 | Employee |
| 3 | Employer |
| 4 | Admin |

### טבלה 3: `Job`

הטבלה המרכזית של ניהול המשרות.

| שם השדה | סוג נתונים | אילוץ | תיאור ותפקיד מפורט |
|---|---|---|---|
| `JobID` | COUNTER / INTEGER | PRIMARY KEY, AUTO_INCREMENT | מזהה ייחודי של המשרה, נוצר אוטומטית על ידי Access (שדה COUNTER). |
| `JobTitle` | VARCHAR(200) | NOT NULL | כותרת המשרה, כגון "מתכנת C# Junior". |
| `JobDescription` | MEMO | NOT NULL | תיאור מפורט של המשרה: תחומי אחריות, דרישות, יתרונות. |
| `JobType` | VARCHAR(50) | NOT NULL | סוג המשרה: "Full-time", "Part-time", "Contract", "Internship", "Remote". |
| `JobStatus` | BIT (YES/NO) | NOT NULL | האם המשרה פעילה (True/1) או סגורה (False/0). משרה סגורה אינה מוצגת בקטלוג. |
| `JobFilter` | MEMO | NULL | שדה חופשי המכיל תגיות, מילות מפתח ופרטים נוספים המשמשים לסינון ולחיפוש. |
| `EmployerID` | VARCHAR(50) | FOREIGN KEY → User.UserID | מזהה המעסיק שפרסם את המשרה. |
| `CountryID` | VARCHAR(50) | FOREIGN KEY → Country.CountryID | מדינת מיקום המשרה. |
| `GenreID` | VARCHAR(50) | FOREIGN KEY → Genre.GenreID | תחום המשרה. |
| `CountryName` | Virtual/JOIN | - | שם הארץ (מתמלא ב-JOIN). |
| `GenreTitle` | Virtual/JOIN | - | שם התחום (מתמלא ב-JOIN). |

**שאילתת SELECT מייצגת:**
```sql
SELECT Job.*, Country.CountryName, Genre.GenreTitle 
FROM (Job LEFT JOIN Country ON Job.CountryID = Country.CountryID) 
LEFT JOIN Genre ON Job.GenreID = Genre.GenreID
```
שאילתה זו משתמשת ב-Nested JOIN (LEFT JOIN בתוך LEFT JOIN), שהוא הפורמט הנדרש ב-Access לשאילתות עם יותר מ-JOIN אחד.

### טבלה 4: `Genre`

טבלת עזר (Lookup Table) עבור תחומי המשרות.

| שם השדה | סוג נתונים | אילוץ | תיאור |
|---|---|---|---|
| `GenreID` | VARCHAR(50) | PRIMARY KEY | מזהה ייחודי של התחום. |
| `GenreTitle` | VARCHAR(100) | NOT NULL | שם התחום: "פיתוח תוכנה", "UX Design", "Marketing", "Finance". |
| `GenreDescription` | MEMO | NULL | תיאור מפורט של תחום זה. |

### טבלה 5: `Country`

טבלת עזר עבור מדינות.

| שם השדה | סוג נתונים | אילוץ | תיאור |
|---|---|---|---|
| `CountryID` | INTEGER | PRIMARY KEY | מזהה מספרי ייחודי של המדינה. |
| `CountryName` | VARCHAR(100) | NOT NULL | שם המדינה (בעברית ו/או באנגלית). |

### טבלה 6: `Notification`

טבלה לאחסון הודעות מערכת הנשלחות על ידי מנהל.

| שם השדה | סוג נתונים | אילוץ | תיאור |
|---|---|---|---|
| `NotificationID` | VARCHAR(50) | PRIMARY KEY | מזהה ייחודי נוצר כ-GUID: `Guid.NewGuid().ToString("N")`. |
| `NotificationText` | MEMO | NOT NULL | תוכן הודעת המערכת. |
| `NotificationDate` | VARCHAR(50) | NOT NULL | תאריך ושעת יצירת ההודעה בפורמט ISO 8601. |

### טבלה 7: `Reviews`

| שם השדה | סוג נתונים | אילוץ | תיאור |
|---|---|---|---|
| `ReviewID` | COUNTER | PRIMARY KEY | מזהה ייחודי אוטומטי. |
| `EmployeeID` | INTEGER | FK → User.UserID | מזהה העובד שכתב את הביקורת. |
| `EmployerID` | INTEGER | FK → User.UserID | מזהה המעסיק עליו נכתבה הביקורת. |
| `RatingData` | INTEGER | NOT NULL | ציון מ-1 עד 5. |

### טבלה 8: `ChatMessages`

| שם השדה | סוג נתונים | אילוץ | תיאור |
|---|---|---|---|
| `MessageID` | COUNTER | PRIMARY KEY | מזהה ייחודי אוטומטי של ההודעה. |
| `SenderID` | VARCHAR(100) | NOT NULL | מזהה המשתמש ששלח את ההודעה. |
| `ReceiverID` | VARCHAR(100) | NOT NULL | מזהה המשתמש שקיבל את ההודעה. |
| `MessageText` | MEMO | NOT NULL | תוכן ההודעה. |
| `SentAt` | VARCHAR(50) | NOT NULL | תאריך ושעת שליחת ההודעה. |

**שאילתת שליפת שיחה בין שני משתמשים:**
```sql
SELECT * FROM ChatMessages 
WHERE (SenderID=@Sender1 AND ReceiverID=@Receiver1) 
   OR (SenderID=@Sender2 AND ReceiverID=@Receiver2) 
ORDER BY MessageID ASC
```
שאילתה זו שולפת את כל ההודעות שעברו בין שני המשתמשים בשני הכיוונים, ממוינות כרונולוגית לפי `MessageID` בסדר עולה.

**שאילתת שליפת שותפי שיחה:**
```sql
SELECT DISTINCT SenderID AS PartnerID FROM ChatMessages WHERE ReceiverID=@UserID1
UNION
SELECT DISTINCT ReceiverID AS PartnerID FROM ChatMessages WHERE SenderID=@UserID2
```
שאילתה זו משתמשת ב-UNION של שתי שאילתות SELECT:
- הראשונה מוצאת את כל השולחים שהמשתמש קיבל מהם הודעה.
- השנייה מוצאת את כל המקבלים שהמשתמש שלח אליהם הודעה.
הכלי `DISTINCT` מבטיח שכל שותף מופיע פעם אחת בלבד.

## 3.3 טבלת קישור

### טבלת `JobApplication` – טבלת קישור חובה

טבלה זו היא ה-**Bridge Table** (טבלת גשר) המיישמת את קשר ה-Many-to-Many בין טבלת ה-Users (עובדים) לטבלת ה-Jobs (משרות). ללא טבלה זו, לא ניתן לייצג את העובדה שמחפש עבודה אחד יכול להגיש מועמדות למשרות רבות, ובמשרה אחת יכולים להיות מועמדים רבים.

| שם השדה | סוג נתונים | אילוץ | תיאור מפורט |
|---|---|---|---|
| `ApplicationID` | COUNTER | PRIMARY KEY | מפתח ראשי אוטומטי ייחודי. Access מגדיל אותו אוטומטית ב-1 לכל INSERT. |
| `JobID` | INTEGER | FOREIGN KEY → Job.JobID | מזהה המשרה. מקשר לטבלת Job. |
| `EmployeeID` | INTEGER | FOREIGN KEY → User.UserID | מזהה מחפש העבודה שהגיש מועמדות. מקשר לטבלת User. |
| `ResumeSnapshot` | MEMO | NOT NULL (Default: '') | צילום מצב של קורות החיים של המועמד **בזמן הגשת המועמדות**. חשוב: גם אם המועמד יעדכן את קורות החיים שלו בעתיד, הרשומה שנשמרה כאן תשמור על הגרסה שהוגשה בפועל. |
| `SubmittedAtUTC` | VARCHAR(50) | NOT NULL | תאריך ושעת הגשת המועמדות בפורמט UTC. נוצר ב-Controller: `DateTime.UtcNow.ToString("yyyy-MM-dd HH:mm:ss")`. |
| `Status` | VARCHAR(20) | NOT NULL (Default: 'Submitted') | הסטטוס הנוכחי של המועמדות: "Submitted" (הוגשה), "Accepted" (התקבל), "Rejected" (נדחה). |

**שאילתת INSERT:**
```sql
INSERT INTO JobApplication (JobID, EmployeeID, ResumeSnapshot, SubmittedAtUTC, Status)
VALUES (@JobID, @EmployeeID, @ResumeSnapshot, @SubmittedAtUTC, @Status)
```

**שאילתת שליפת מועמדויות לפי מחפש עבודה:**
```sql
SELECT * FROM JobApplication WHERE EmployeeID=@EmployeeID
```
שאילתה זו משמשת את `GetJobHistory` ב-`UserController` לשליפת כל המשרות אליהן הגיש מחפש עבודה מועמדות.

**שאילתת שליפת מועמדויות לפי משרה:**
```sql
SELECT * FROM JobApplication WHERE JobID=@JobID ORDER BY SubmittedAtUTC DESC
```
שאילתה זו משמשת את `Applications` ב-`EmployerController` לצפייה ברשימת המועמדים.
