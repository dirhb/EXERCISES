1. מבוא

רקע ומטרות המערכת

בעולם התעסוקה המודרני, מציאת משרה מתאימה וגיוס עובדים איכותיים הם אתגרים מורכבים הדורשים כלים טכנולוגיים מתקדמים. פרויקט זה מציג את מערכת JobFindUltimate, פלטפורמת אינטרנט מקיפה המתווכת בין מחפשי עבודה (Employees) לבין מעסיקים (Employers) ומנוהלת על ידי מנהלי מערכת (Admins).

מטרת על:

יצירת פלטפורמה אמינה, מהירה וידידותית למשתמש המאפשרת למחפשי עבודה למצוא משרות רלוונטיות, להגיש מועמדות ולעקוב אחר ההיסטוריה התעסוקתית שלהם, בעוד שמעסיקים יכולים לפרסם משרות, לנהל מועמדים ולעדכן פרטי העסקה ושכר בצורה יעילה ומאובטחת.

מטרות נלוות:

  • פיתוח מנגנון תקשורת מבוסס צ'אט מובנה בין מעסיקים למועמדים לשיפור תהליך הגיוס.
  • הגדרת בקרת גישה וסמכויות מבוססות תפקיד (Role-Based Authorization) להפרדה מוחלטת בין פנל המנהל, המעסיק ומחפש העבודה.
  • מתן התראות אוטומטיות למשתמשים על שינויים ועדכונים באתר.

קהל היעד וגבולות המערכת

המערכת מיועדת לשלושה קהלי יעד מרכזיים:

  1. מחפשי עבודה (Employees): אנשים המחפשים משרות פנויות, המעוניינים לבנות קורות חיים מקוונים ולהגיש מועמדות למשרות שונות.
  2. מעסיקים (Employers): חברות ומגייסים המעוניינים לפרסם משרות, לצפות בקורות החיים של המועמדים, ולנהל את תהליך הקבלה.
  3. מנהלי מערכת (Admins): גורם מפקח האחראי על ניטור תוכן המשרות, מחיקת משתמשים פוגעניים או משרות לא רלוונטיות, ושליחת הודעות מערכת גלובליות.

גבולות המערכת:

המערכת פועלת כרשת סגורה שבה המעסיק ומחפש העבודה פועלים תחת ישות ניהולית אחת. תהליך הריאיון הראשוני יכול להתבצע בצ'אט הפנימי, אך הליכים משפטיים וחוזיים של קבלה לעבודה מתבצעים מחוץ למערכת (המערכת מספקת רק את הפלטפורמה לאיתור המועמדים ועדכון הסטטוס הראשוני).

סביבות פיתוח וטכנולוגיות

הפרויקט ממומש בארכיטקטורת Client-Server מודרנית המופרדת לשני פרויקטים עצמאיים:

  • צד שרת (Backend - Web Service): נבנה באמצעות טכנולוגיית ASP.NET Core Web API בשפת #C. השרת מנהל את הגישה למסד הנתונים ומספק קצוות קצה (API Endpoints) בפורמט JSON.
  • צד לקוח (Frontend - Web Client): נבנה בטכנולוגיית ASP.NET Core MVC המבצעת קריאות HTTP א-סינכרוניות לשרת ה-API ומציגה דפי אינטרנט דינמיים המעוצבים ב-HTML, CSS ו-JavaScript.
  • מסד הנתונים (Database): מסד נתונים יחסי מסוג Microsoft Access (.accdb), הגישה אליו מתבצעת באמצעות פרוטוקול OleDb בשילוב מנגנון פרמטרים להגנה מפני SQL Injection.
  • סביבת הפיתוח המרכזית: Microsoft Visual Studio 2022.

אתגרים ופתרונות

  • עבודה אסינכרונית וביצועים: כדי למנוע חסימה של ה-Thread הראשי בעת ביצוע קריאות רשת מהלקוח לשרת ובעת גישה למסד הנתונים, השרת והלקוח ממומשים במלואם בעזרת מנגנון async ו-await ב-#C.
  • ניהול נעילות במסד הנתונים (MS Access): מסד הנתונים MS Access אינו מיועד לעומסים מרובי משתמשים במקביל. הפתרון היה מימוש מחלקת עזר מאוחדת לניהול החיבור (DBHelperOledb) וניהול מחזור החיים של החיבורים בצורה אחראית (פתיחה וסגירה מיידית של חיבור ה-OleDb בכל פעולה בקריאות השרת).
  • הגנה על נתונים רגישים: אבטחת סיסמאות משתמשים במסד הנתונים והגנה על שאילתות מפני פריצות SQL Injection על ידי שימוש ב-Parameterized Queries בכל מחלקות ה-Repository.

2. ניתוח המערכת

מסמך ייזום ותיאור המצב הקיים

במצב הקיים, חיפוש משרות וניהול מועמדים מתבצע לרוב באופן לא ריכוזי (שליחת קבצי PDF בדוא"ל, מעקב בטבלאות אקסל ידניות, וכדומה). תהליך זה מוביל לאובדן מידע, חוסר יכולת לעקוב אחר סטטוס המועמדות, ועיכובים בתקשורת בין המגייס למועמד.

מערכת JobFindUltimate מיישמת מסמך ייזום המגדיר מערכת אחודה שבה כל מחזור החיים של המועמדות - החל מהחיפוש, דרך הסינון, ההגשה ועד לסטטוס הסופי והצ'אט - מרוכזים תחת קורת גג אחת דיגיטלית.

עץ תהליכים

המערכת מורכבת מהתהליכים הראשיים הבאים:

  1. רישום והתחברות (Authentication): אימות משתמשים, זיהוי סוג המשתמש (מחפש עבודה/מעסיק/מנהל).
  2. ניהול משרות (Job Management): פרסום משרה, עדכון משרה, מחיקת משרה, וקטלוג משרות לפי תחומים (Genres) ומיקומים.
  3. ניהול מועמדויות (Application Tracking): הגשת מועמדות למשרה (צירוף קורות חיים), צפייה ברשימת המועמדים, עדכון סטטוס מועמדות ("Submitted", "Accepted", "Rejected").
  4. ניהול עובדים ושכר (Employee & Salary Management): שינוי שכר המועמד על ידי המעסיק, פיטורין/הפסקת העסקה.
  5. תקשורת (Chat & Notifications): ניהול צ'אט חי בין מועמדים למעסיקים ושליחת התראות גלובליות על ידי מנהל המערכת.

דיאגרמות Use Case ו-DFD

בשלב זה של הפרויקט, הוגדרו תרחישי השימוש (Use Cases) ודיאגרמות זרימת הנתונים (DFD) לכל סוג משתמש:

מחפש עבודה ואורח (Guest & Employee)

Use Case: האורח יכול לצפות בדף הבית ובקטלוג המשרות, להפעיל סינונים (מיקום, סוג משרה, שכר, ניסיון, גמישות) ולהירשם למערכת. מחפש העבודה המחובר (Employee) יכול להשתמש בכל כלי הניווט, לצפות בהיסטוריית הגשות המשרות שלו, לעדכן את קורות החיים שלו באופן מקוון, ולהגיש מועמדות למשרה בקטלוג.

זרימת נתונים (DFD): האורח שולף נתונים מקטלוג המשרות (השואב ממאגר D1 - Jobs) ומבצע רישום (המעדכן את מאגר D3 - Employee). מחפש העבודה מעדכן את קורות החיים (במאגר D5 - Resume) ומגיש מועמדות, תהליך היוצר רשומה חדשה במאגר D4 - Application.

מעסיק (Employer)

Use Case: המעסיק יכול לבצע ארבע פעולות ראשיות: הוספת משרה (תוך קביעת קטגוריה, שכר בסיס, ניסיון נדרש, מיקום, והטבות), ניהול משרות (עדכון או מחיקה), צפייה במועמדויות (כולל סינון לפי קורות חיים), וניהול עובדים (עדכון שכר או פיטורין).

זרימת נתונים (DFD): הוספת המשרות מעדכנת את מאגר D1 - Jobs. צפייה במועמדויות מושכת נתונים ממאגר המועמדויות D4 - Application ומאגר העובדים D3 - Employee. ניהול עובדים מעדכן את השכר ב-D3 או מוחק מועמדות מ-D4.

מנהל מערכת (Admin)

Use Case: מנהל המערכת יכול לנהל משרות גלובלית (ולמחוק משרות מיושנות או לא ראויות), למחוק או לחסום משתמשים בעייתיים, לסקור את כל המועמדויות במערכת, ולשלוח התראות גלובליות לכל המשתמשים על שינויים או עדכונים באתר.

זרימת נתונים (DFD): מחיקת משרות מעדכנת את מאגר D1 - Jobs. מחיקת משתמשים מעדכנת את מאגר D2 - Users. יצירת התראה מעדכנת את מאגר D4 - Notifications.

3. בסיס הנתונים (Database)

דיאגרמת קשרי גומלין (DSD / ERD)

בסיס הנתונים מבוסס על מודל טבלאות יחסי המקושרות ביניהן באמצעות מפתחות זרים (Foreign Keys). הקשרים מבוססים על ישויות מרכזיות: משתמשים, תפקידים, משרות, מועמדויות, ארצות, תחומי משרה, הודעות צ'אט, ביקורות והתראות.

טבלאות ראשיות וטבלאות קישור

1. טבלת משתמשים (User / Users)

שם השדה סוג נתונים מפתח תיאור ותפקיד
UserID VARCHAR ראשי (PK) מזהה ייחודי של המשתמש (תעודת זהות)
FirstName VARCHAR - שם פרטי
LastName VARCHAR - שם משפחה
UserName VARCHAR - שם המשתמש המשמש להתחברות (ייחודי)
Password VARCHAR - סיסמת המשתמש מוצפנת/מאובטחת
UserTypeID INTEGER זר (FK) מזהה תפקיד (מנהל/עובד/מעסיק)
ResumeText MEMO/TEXT - טקסט קורות החיים של מחפש העבודה

2. טבלת משרות (Jobs)

שם השדה סוג נתונים מפתח תיאור ותפקיד
JobID VARCHAR ראשי (PK) מזהה ייחודי של המשרה
JobTitle VARCHAR - כותרת המשרה
JobDescription MEMO/TEXT - תיאור ודרישות המשרה
EmployerID VARCHAR זר (FK) מזהה המעסיק שפרסם את המשרה
GenreID VARCHAR זר (FK) מזהה תחום המשרה (קטגוריה)

3. טבלת הגשת מועמדות (JobApplication - טבלת קישור חובה)

טבלה זו משמשת כטבלת קישור (Many-to-Many Bridge Table) המקשרת בין טבלת המשתמשים (העובדים) לטבלת המשרות. היא מתעדת את אירוע הגשת המועמדות.

שם השדה סוג נתונים מפתח תיאור ותפקיד
ApplicationID COUNTER ראשי (PK) מזהה ייחודי של הגשת המועמדות (רץ אוטומטית)
JobID INTEGER זר (FK) מזהה המשרה שאליה הוגשה המועמדות
EmployeeID INTEGER זר (FK) מזהה העובד שהגיש את מועמדותו
ResumeSnapshot MEMO - צילום מצב של קורות החיים של המועמד בעת ההגשה
Status VARCHAR - סטטוס המועמדות הנוכחי ("Submitted", "Accepted", "Rejected")

4. מימוש הפרויקט (צד שרת וצד לקוח)

מימוש צד שרת (Web Service API)

השרת ממומש באמצעות ארכיטקטורת REST API מבוססת בקרים (Controllers) המזריקים מחסני נתונים (Repositories) דרך יחידת עבודה (Unit of Work). כל שאילתות ה-SQL ממומשות בצורה מאובטחת ומנוהלות א-סינכרונית.

קוד מחלקת המודל Job.cs:

using System.ComponentModel.DataAnnotations;

namespace JobModels
{
    public class Job
    {
        [Required]
        public string? JobID { get; set; }
        public string? JobTitle { get; set; }
        public string? JobDescription { get; set; }
        [Required]
        public string? JobType { get; set; }
        [Required]
        public bool? JobStatus { get; set; }
        public string? JobFilter { get; set; }
        public string? EmployerID { get; set; }
        public string? CountryID { get; set; }
        public string? GenreID { get; set; }
        public string? CountryName { get; set; }
        public string? GenreTitle { get; set; }
    }
}

קוד מחסן הנתונים UserRepository.cs (קטע מתוך עדכון קורות חיים):

מציג שימוש בפרמטריזציה להגנה מפני SQL Injection:

public bool UpdateResume(string userId, string resumeText)
{
    string sql = "UPDATE [User] SET ResumeText=@ResumeText WHERE UserID=@UserID";
    
    this.helperOleDb.ClearParameters();
    this.helperOleDb.AddParameters("@ResumeText", resumeText);
    this.helperOleDb.AddParameters("@UserID", userId);
    
    return this.helperOleDb.Update(sql) > 0;
}

מימוש צד לקוח (MVC Web Client)

צד הלקוח פונה בצורה א-סינכרונית לשרת ה-API באמצעות מחלקת עזר גנרית ApiClient<T>.

קוד מתוך ApiClient.cs (קריאת GET א-סינכרונית):

public async Task<T> GetAsync()
{
    using (HttpRequestMessage httpRequest = new HttpRequestMessage())
    {
        httpRequest.Method = HttpMethod.Get;
        httpRequest.RequestUri = this.uriBuilder.Uri;
        using (HttpResponseMessage httpResponse = await this.httpClient.SendAsync(httpRequest))
        {
            if (httpResponse.IsSuccessStatusCode == true)
            {
                string result = await httpResponse.Content.ReadAsStringAsync();
                JsonSerializerOptions jsonSerializerOptions = new JsonSerializerOptions();
                jsonSerializerOptions.PropertyNameCaseInsensitive = true;
                return JsonSerializer.Deserialize<T>(result, jsonSerializerOptions);
            }
            return default(T);
        }
    }
}

מנגנוני אבטחה והרחבות

הגנה מ-SQL Injection באמצעות Parameterized Queries

כל שאילתות מסד הנתונים בפרויקט מבוצעות על ידי הזרקת פרמטרים ייעודיים לתוך OleDbCommand. הפרמטרים מנוקים ומתווספים באופן הבא: this.helperOleDb.AddParameters("@UserName", userName). שיטה זו מפרידה לחלוטין בין הקוד של פקודת ה-SQL לבין הנתונים המוזנים על ידי המשתמש, ומגינה על המערכת מפני ניסיונות פריצה ושיבוש שאילתות.

5. מדריך למשתמש

מערכת JobFindUltimate מגדירה ממשק שונה לחלוטין עבור שלושה סוגי משתמשים עיקריים:

אורח ומחפש עבודה (Guest & Employee)

  • רישום והתחברות: לחיצה על כפתור המשתמש בפינה השמאלית העליונה פותחת את עמוד ההתחברות. משתמש חדש יכול להירשם על ידי בחירת תפקיד (מחפש עבודה או מעסיק) והזנת פרטים אישיים.
  • קטלוג משרות: ניווט לקטלוג המשרות מציג משרות פנויות. ניתן לסנן את המשרות בעזרת סרגל הסינון הימני לפי ארץ (Country) ותחום (Genre).
  • הגשת מועמדות: עובד מחובר יכול ללחוץ על כפתור הגשת המועמדות בכרטיס המשרה כדי לשלוח מיידית את קורות החיים המעודכנים שלו.
  • ניהול קורות חיים: בעמוד הפרופיל שלו, העובד יכול לערוך ולשמור את קורות החיים שלו בתיבת הטקסט המקוונת.

מעסיק (Employer)

  • פרסום משרה חדשה: מעסיק יכול לגשת לטופס "פרסום משרה" ולהזין את כותרת המשרה, התיאור, השכר המוצע, ודרישות הניסיון.
  • ניהול מועמדים: בלשונית "Applicants" (מועמדים), המעסיק יכול לצפות בכל האנשים שהגישו מועמדות למשרות שלו, לקרוא את קורות החיים שלהם ולשנות את סטטוס המועמדות ל-"Accepted" או "Rejected".
  • ניהול עובדים ושכר: במסך המועמדים שהתקבלו, המעסיק יכול לעדכן את שכרם ישירות או להפסיק את העסקתם ("פיטורין").

מנהל מערכת (Admin)

  • ניקוי תוכן: המנהל יכול למחוק משרות מיושנות או משתמשים המפרים את כללי הקהילה.
  • שליחת הודעות מערכת: המנהל יכול להזין תוכן הודעה שיופץ כהתראה לכל המשתמשים במערכת בעת כניסתם.

6. סיכום אישי ורפלקציה

העבודה על פרויקט JobFindUltimate הייתה חוויה מלמדת ומאתגרת במיוחד. הפרויקט אפשר לי לחבר בין מספר תחומים מרכזיים בהנדסת תוכנה: תכנון בסיס נתונים יחסי, יצירת שירותי רשת (Web Services) א-סינכרוניים, ובניית ממשק משתמש דינמי בטכנולוגיית MVC.

הצלחת הפרויקט נבעה בעיקר מתכנון מוקדם נכון של בסיס הנתונים ודיאגרמות ה-ERD וה-DFD, אשר מנעו תקלות רבות בשלבי הפיתוח המאוחרים. האתגר המרכזי שעימו התמודדתי היה מניעת חסימות ונעילות במסד הנתונים MS Access בעת ביצוע קריאות רשת א-סינכרוניות במקביל. פתרתי בעיה זו על ידי ניהול קפדני של מחזור החיים של חיבור ה-OleDb דרך יחידת העבודה (Unit of Work), תוך הקפדה על סגירת החיבור בסיום כל פעולה.

תובנה מרכזית מהתהליך היא חשיבותו של קוד אסינכרוני נכון. שימוש ב-async ו-await מאפשר למערכת לעבוד בצורה חלקה וריספונסיבית יותר. בראייה לאחור, אם היו לי משאבים נוספים, הייתי בוחר להחליף את מסד הנתונים MS Access במסד נתונים מתקדם יותר כגון SQL Server או PostgreSQL כדי לאפשר תאימות מלאה וביצועים גבוהים יותר בסביבה מרובת משתמשים.

8. נספחים וקוד המקור

בנספח זה מובא קוד האתחול הראשי של השרת ושל הלקוח בפרויקט:

קובץ אתחול השרת Program.cs (בפרויקט JobWebService):

using JobWebService.ORM;
using JobWebService.ORM.Repositories;
using JobModels;

namespace JobWebService
{
    public class Program
    {
        public static void Main(string[] args)
        {
            var builder = WebApplication.CreateBuilder(args);

            builder.Services.AddControllers();
            builder.Services.AddSingleton<DBHelperOledb>();
            builder.Services.AddSingleton<ModelCreators>();

            builder.Services.AddScoped<IRepository<User>, UserRepository>();
            builder.Services.AddScoped<IRepository<Job>, JobRepository>();
            builder.Services.AddScoped<IRepository<Genre>, GenreRepository>();
            builder.Services.AddScoped<IRepository<Review>, ReviewRepository>();
            builder.Services.AddScoped<IRepository<Notification>, NotificationRepository>();
            builder.Services.AddScoped<IRepository<Country>, CountryRepository>();
            builder.Services.AddScoped<IRepository<Education>, EducationRepository>();
            builder.Services.AddScoped<IRepository<EducationType>, EducationTypeRepository>();
            builder.Services.AddScoped<IRepository<UserType>, UserTypeRepository>();
            builder.Services.AddScoped<IRepository<ChatMessage>, ChatMessageRepository>();
            
            builder.Services.AddEndpointsApiExplorer();
            builder.Services.AddSwaggerGen();

            var app = builder.Build();

            if (app.Environment.IsDevelopment())
            {
                app.UseSwagger();
                app.UseSwaggerUI();
            }

            app.UseAuthorization();
            app.MapControllers();
            app.Run();
        }
    }
}

קובץ אתחול הלקוח Program.cs (בפרויקט JobWebApp):

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllersWithViews();
builder.Services.AddDistributedMemoryCache();

builder.Services.AddSession(options =>
{
    options.IdleTimeout = TimeSpan.FromMinutes(30);
    options.Cookie.HttpOnly = true;
    options.Cookie.IsEssential = true;
});

var app = builder.Build();

if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler("/Guest/Error");
    app.UseHsts();
}

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();

app.UseSession();
app.UseAuthorization();
app.MapStaticAssets();

app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Guest}/{action=Home}/{id?}");

app.Run();
